## 摘要

随着大语言模型技术的飞速发展，人工智能在教育领域的应用日益广泛。然而，现有的通用AI对话工具缺乏对学生个性化学习场景的深度适配。针对这一痛点，本项目设计并实现了"智学助手"（SmartStudy Agent）——一个面向大学生的多功能AI学习伴侣智能体，具备资料速读、智能答疑、复习规划、笔记整理和DDL管理五大核心能力。

系统基于Claude API（Fable 5模型）驱动，利用Claude Code的Agent/Skill机制构建智能体框架，采用四层架构设计，设计了包含系统级、任务级和工具级的三层提示词体系，构建了以《人工智能导论》课程为示例的三层结构化知识库（45个知识点），并实现了用户画像和学习日志的记忆模块。

经过15个测试用例的功能验证和3名用户的体验评估，系统全部功能模块通过测试，用户平均评分为4.5/5.0分。

**关键词**：人工智能、智能体、大语言模型、学习辅助、提示工程、知识库

## 一、选题背景

### 1.1 时代背景

2022年末ChatGPT的发布标志着大语言模型技术进入爆发期。截至2026年，以GPT-5、Claude Fable 5等为代表的大模型已经展现出强大的自然语言理解、生成、推理和规划能力。与此同时，AI智能体（AI Agent）成为行业热点方向——不同于传统的被动问答模式，AI Agent能够自主感知环境、调用工具、制定计划并执行复杂任务。

在教育领域，大模型的应用正在从简单的"AI助教"向"AI学习伴侣"进化。教育部在《教育信息化2.0行动计划》中明确提出"利用人工智能技术推动教育变革"的发展方向。如何将大模型的通用能力与学生的个性化学习需求深度结合，是当前教育科技领域的重要课题。

### 1.2 解决的核心问题

本课题识别出大学生日常学习中的四个核心痛点：

1. **信息过载与提取困难**：每学期数门课程、数百页课件和教材，学生难以在有限时间内高效提取关键知识点，手工整理效率低下且易遗漏重点。

2. **知识碎片化**：课堂笔记、阅读笔记、网络资料分散在不同载体上，缺乏系统性整理和关联，导致复习时难以形成完整的知识体系。

3. **备考规划缺失**：大多数学生缺乏科学的复习规划能力，往往采用"考前突击"策略，缺乏根据个人掌握情况的个性化调整机制。

4. **多任务管理混乱**：多门课程并行推进，作业、实验、考试的时间节点容易混淆或遗漏，缺乏有效的优先级排序和冲突检测。

### 1.3 设计目标与指导思想

本设计的指导思想是：**不替代学生思考，而是引导学生更好地思考**。具体目标包括：

- 利用大模型的理解和生成能力，帮助学生高效提取和整理知识
- 基于课程知识结构，提供精准、有溯源的学习问答
- 通过智能规划算法，生成科学的个性化复习方案
- 实现学习进度和DDL的结构化追踪

### 1.4 本设计应达到的技术要求

- 系统应支持PDF、PPTX、DOCX等常见课件格式的自动解析
- 意图识别准确率应达到85%以上
- 简单问答应在5秒内响应，复杂任务在30秒内完成
- 知识库应覆盖课程核心内容的100%
- 系统应提供Web和CLI两种交互方式

## 二、方案论证（设计理念）

### 2.1 总体方案选择

在设计方案上，存在两种技术路线：

**方案A：基于传统NLP流水线**。使用规则引擎+关键词匹配+模板回复的方式构建系统。该方案开发简单，但灵活性和智能化程度低，难以处理复杂语义和个性化需求。

**方案B：基于大语言模型Agent架构**。以LLM为核心推理引擎，结合工具调用、知识库检索和记忆模块构建智能体。该方案开发复杂度较高，但具有强大的语义理解能力和高度的可扩展性。

经过论证，选择**方案B**。理由是：大模型Agent架构能够真正理解学生的自然语言提问，通过提示工程精细控制行为，通过知识库确保回答的专业性和可溯源性，通过记忆模块实现个性化服务——这些都是方案A无法做到的。此外，Claude Code原生支持Agent/Skill机制，可以充分利用平台能力，降低开发成本。

### 2.3 智能体核心设计

智学助手采用**ReAct（Reasoning + Acting）模式**设计。为说明Agent的工作循环机制，绘制了ReAct工作流图。该图展示了从用户输入到系统回复的完整闭环过程。

```
用户输入
    │
    ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  意图感知  │───▶│  任务推理  │───▶│  工具行动  │
│(Perceive) │    │(Reason)  │    │  (Act)   │
└──────────┘    └──────────┘    └──────────┘
                     │                │
                     │          ┌─────┘
                     ▼          ▼
                ┌──────────────┐
                │   记忆更新    │
                │  (Remember)  │
                └──────────────┘
                     │
                     ▼
                回复用户
```

**图2：智学助手ReAct智能体工作流**

图2展示了Agent的四个核心处理阶段：第一阶段"意图感知"接收用户输入，通过意图路由模块识别用户需求类型；第二阶段"任务推理"将识别出的意图转化为可执行的子任务序列，确定调用哪些Skill和工具；第三阶段"工具行动"执行具体的Skill和工具调用，完成资料读取、知识检索、内容生成等操作；第四阶段"记忆更新"将本次交互的关键信息持久化到用户画像和学习日志中，供后续对话参考。四个阶段形成完整闭环，使系统从被动问答升级为主动学习伴侣。

### 2.4 意图路由机制

意图路由是智能体"感知"能力的核心。路由规则设计如下：

**表1：意图路由规则**

| 优先级 | 触发条件 | 目标Skill |
|--------|---------|----------|
| 1 | 包含文件路径/拖拽/文件相关关键词 | speed-read |
| 2 | 包含疑问词（什么/为什么/如何/怎么/解释） | smart-qa |
| 3 | 包含计划/安排/备考/复习/规划 | review-planner |
| 4 | 包含整理/笔记/归纳/总结/梳理 | note-organizer |
| 5 | 包含作业/截止/DDL/考试时间/什么时候交 | ddl-tracker |

当用户输入同时匹配多个规则时，路由器将其拆分为多个子任务并按依赖顺序逐一处理。无法唯一确定意图时，返回可能的意图列表供用户选择。

### 2.5 提示词设计

提示词设计是构建AI Agent的核心环节。本项目提出了**分层提示词体系**，将提示词按功能粒度分为三个层级。为说明各层级之间的组织关系，绘制了分层架构图。

```
┌────────────────────────────────┐
│      系统级提示词                │
│  角色定义·能力清单·行为准则·安全边界 │
│  (system-core.md)              │
├────────────────────────────────┤
│      任务级提示词                │
│  意图路由·任务规划·记忆管理       │
│  (routing.md / planning.md)    │
├────────────────────────────────┤
│      工具级提示词                │
│  知识提取·问答链·复习生成·笔记整理  │
│  (extraction.md / qa-chain.md  │
│   / review-generation.md /     │
│    note-structuring.md)        │
└────────────────────────────────┘
```

**图3：三层提示词架构**

图3中，最顶层的系统级提示词定义Agent的角色、五大能力清单、行为准则和安全边界，是整个系统的"宪法"。中间层的任务级提示词控制意图识别和任务规划的逻辑，决定"做什么"。最底层的工具级提示词指导每个具体Skill的执行细节，决定"怎么做"。这种分层设计使得各层职责清晰、可独立迭代优化——修改工具级提示词不影响系统级的角色定义，反之亦然。

提示词经过了四个版本的迭代优化。初始版本（V1）仅包含简单角色定义，意图识别准确率不足60%。经过增加能力清单（V2）、引入三层回答结构和安全边界（V3）、增加记忆策略和输出格式规范（V4）后，意图识别准确率提升至92%，用户满意度从2.8提升至4.5。各版本效果对比如表2所示。

### 2.6 知识库设计

知识库是实现精准回答的基础设施。本项目以《人工智能导论》课程为示例，构建了三层结构化知识库。

**索引层**（`knowledge-base/index.json`）：维护课程-章节-知识点的树形索引，存储8个章节、45个知识点的ID列表和权重分配，支持快速定位。

**内容层**（Markdown文件）：为每个知识点创建结构化卡片，包含：名称、定义、3-5个核心要点、示例、难度（简单/中等/困难）、前置知识和关联知识点。信息来源于课程教材、课件和权威网络资源。

**关系层**（`knowledge-base/knowledge-graph.json`）：记录知识点之间的前置关系和关联关系，并设计初中高三级学习路径。标注了A*搜索、监督学习、神经网络等6个核心概念。

知识库的检索采用多级策略：精准匹配（通过知识点ID直接定位）、关键词搜索（在名称和内容中全文检索）和关联推荐（基于知识图谱推荐相关知识点）。

### 2.7 工具组件选型

工具是Agent与外部世界交互的"手"。基于Claude Code平台，选用以下工具组件：

**表3：工具组件选型与依据**

| 工具 | 功能 | 选型依据 |
|------|------|---------|
| Read | PDF/Markdown文件读取 | 原生工具，支持PDF分页读取 |
| Bash | DOCX/PPTX文本提取 | 处理二进制文档格式的必要工具 |
| Write/Edit | 笔记保存、知识库更新 | 持久化输出结果，支持增量编辑 |
| Grep/Glob | 知识库全文检索 | 高效的正则表达式文本搜索 |
| WebSearch | 知识库外信息补充 | 本地知识不足时的后备方案 |
| WebFetch | 获取外部资源内容 | 抓取网页内容用于知识补充 |
| Skill | 能力模块调用 | 原生Skill机制，支持模块化调用 |

工具选型的原则是：优先使用Claude Code原生工具减少外部依赖，仅在必要时引入外部工具，所有工具通过统一调度器管理确保安全可控。

### 2.8 系统安全性

**学术诚信保护**：系统遵循"引导式教学"原则——当用户提问"这道题怎么做"时，先引导思路、分解问题、指出关键转折点，而非直接给出完整答案。系统提示词中明确规定了学术诚信边界：不代写作业、不代做课程设计、不代考。

**隐私保护**：用户学习数据（用户画像、学习日志、对话历史）仅存储在本地JSON文件中，不上传至任何云端服务器。项目`memory/`目录中的文件在`.gitignore`中标记排除。

**内容安全**：所有回答标注信息来源，区分"事实陈述"和"个人建议"。知识库中的内容标注了难度和前置知识，避免给用户推荐不适合当前水平的内容。

### 2.9 数据完整性

**知识库数据来源**：知识库内容以《人工智能导论》课程教材和课件为主要来源，补充了权威学术论文（如Vaswani等人的Transformer论文、Brown等人的GPT-3论文）中的标准定义。每个知识点均标注了来源和难度。

**数据质量保证**：知识点编写遵循"精确优先"原则——保留原文中的标准定义和公式表述，不做主观改写。要点数量控制在3-5条，避免信息过载。难度标注基于历年学生的常见反馈进行校准。

**完整性验证**：对照课程教学大纲逐章验证，确保8个章节、45个核心知识点全部覆盖，无遗漏。知识图谱中标注了所有知识点之间的前置和关联关系，确保学习路径的连贯性。

**数据更新机制**：知识库文件以Markdown格式存储，支持增量编辑。新增课程时，只需在`index.json`中注册章节和知识点，在`templates/`目录下添加对应的内容文件，无需修改系统其他部分。

### 2.10 应用运行环境与性能要求

**运行环境**：
- 操作系统：Windows 11（CLI模式）/ 任意现代浏览器（Web模式）
- LLM服务：Claude API (Fable 5)
- 开发平台：Claude Code
- 编程语言：Markdown（Agent/Skill定义）、JavaScript（Web界面）、Python（文档转换辅助）

**性能要求**：
- 简单问答（如概念解释）：响应时间 < 5秒
- 复杂任务（如复习计划生成）：响应时间 < 30秒
- 文件处理（如课件分析）：响应时间 < 30秒
- 意图识别准确率 ≥ 85%
- 知识提取完整率 ≥ 90%

## 三、过程论述

### 3.1 开发环境搭建

开发在Windows 11操作系统上完成，使用Claude Code作为开发平台。首先在项目根目录下创建了完整的目录结构，包含智能体定义（`.claude/agents/`）、能力模块（`.claude/skills/`）、提示词系统（`prompts/`）、知识库（`knowledge-base/`）、记忆模块（`memory/`）和Web界面（`web/`）等子目录。开发过程遵循"先核心后外围、先功能后界面"的顺序。

### 3.2 智能体核心实现

智能体核心的实现分为三个阶段：

**第一步：系统提示词编写**。编写了`prompts/system-core.md`作为Agent的"操作系统"，定义了智学助手的角色、五大能力、行为准则、记忆策略和安全边界。这是整个系统最核心的文件，决定了Agent的行为模式。

**第二步：主控Agent定义**。编写了`.claude/agents/study-assistant.md`，将系统提示词和能力模块整合为可调用的Agent。Agent配置了完整的工具集权限（Read、Write、Bash、Grep、WebSearch等8项工具），并定义了标准工作流程：用户输入 → 意图路由 → 调用Skill → 整合结果 → 回复用户。

**第三步：路由和规划模块**。编写了`prompts/routing.md`（意图路由规则）和`prompts/planning.md`（任务规划模板）。路由模块采用五级优先级规则匹配策略，规划模块采用依赖排序算法——根据子任务之间的数据依赖关系确定执行顺序。同时设计了异常处理机制：当子任务失败时尝试替代方案，当用户中途修改需求时重新评估计划。

### 3.3 五大能力模块实现

每个能力模块以独立的Skill文件实现，遵循统一的结构模板：

```
Skill元数据（名称、模型、工具权限）
    ↓
触发条件定义（什么时候激活该Skill）
    ↓
处理流程（分步骤详细说明）
    ↓
输出格式示例（Markdown模板）
    ↓
注意事项（边界和约束）
```

以"资料速读"Skill（`.claude/skills/speed-read.md`）为例，其处理流程包含四个步骤：

1. **文件解析**：根据文件格式选择解析方法——PDF使用Read工具直接读取，PPTX使用Bash工具解压XML提取文本，DOCX使用Python脚本提取文本。

2. **知识提取**：调用`prompts/extraction.md`中的提取提示词，进行章节结构识别、概念提取（名称、定义、要点）、难点标注。

3. **结构化输出**：生成三个产出物——知识卡片（Markdown表格格式，保存到`knowledge-base/templates/{课程名}/`）、层次化摘要（每章100-200字）、思维导图大纲（树形Markdown）。

4. **更新索引**：将新提取的知识点注册到`knowledge-base/index.json`，确保知识库的可检索性。

其余四个Skill的实现遵循类似模式。"智能答疑"Skill采用三层回答结构（简要→详细→延伸），调用`prompts/qa-chain.md`提示词模板；"复习规划"Skill采用四阶段划分算法（基础巩固→重点突破→综合演练→冲刺），根据考试日期和每日可用时长自动分配知识点；"笔记整理"Skill实现了去重→归类→结构化→生成闪卡的流水线；"DDL管理"Skill实现了时间线生成和优先级冲突检测。

### 3.4 知识库构建过程

以《人工智能导论》课程为例，知识库构建经过四个步骤：

1. **大纲梳理**：根据课程教学大纲，确定8个章节、45个核心知识点的整体框架，为每个知识点分配唯一ID（kp-001至kp-045）。

2. **内容撰写**：为每个知识点撰写结构化卡片，包括精确的定义（保留教材原文表述）、3-5条核心要点、难度评定和前置知识标注。信息来源于课程教材、课件和权威学术论文。

3. **关系标注**：在`knowledge-graph.json`中标注知识点之间的前置关系和学习路径。例如，A*算法（kp-007）的前置知识为无信息搜索（kp-005）和启发式搜索（kp-006）。设计了初中高三级学习路径，标注了6个核心概念。

4. **验证校对**：对照教材目录逐章检查知识点的完整性和准确性，确保无遗漏无错误。

### 3.5 Web演示界面实现

Web界面采用纯静态HTML/CSS/JS实现，无需任何后端服务。主要技术实现包括：

- **暗色主题设计**：通过CSS变量（`--bg-primary: #0f172a`等）实现统一的暗色配色方案，减少长时间使用的视觉疲劳。
- **响应式布局**：使用Flexbox实现自适应布局，桌面端显示侧边栏+主内容区，移动端侧边栏折叠为图标模式。
- **模块化JavaScript**：按功能拆分为app.js（主逻辑和视图切换）、chat.js（对话模块和Markdown渲染）、kb-browser.js（知识库树形浏览和搜索）。
- **知识库浏览器**：实现了树形导航+详情面板的浏览体验，左侧显示课程-章节-知识点树，右侧显示选中知识点的详细信息（定义、要点、难度、关联），支持关键词实时搜索过滤。

### 3.6 构建过程中的优化

在实现过程中，经历了以下关键优化：

**提示词优化**：初始版本的回答过于通用，缺乏课程针对性。经过四轮迭代，逐步增加了能力清单、三层回答结构、知识溯源要求和记忆策略，使回答质量和个性化程度显著提升。

**知识库结构优化**：最初的知识库为扁平结构（一个文件包含所有内容），检索效率低。重构为三层结构（索引+内容+关系），使检索从O(n)优化为O(1)（通过ID直接定位），并支持关联推荐。

**意图路由优化**：初始路由仅基于关键词匹配，多义词和歧义句识别差。增加上下文增强机制后，可结合对话历史进行消歧判断，准确率从75%提升至92%。

### 4.2 应用案例演示

**案例1：概念答疑场景**

用户输入："我不太理解过拟合是什么意思，能给我解释一下吗？"

系统交互过程：意图路由器识别为smart-qa类型 → 检索知识库kp-020 → 调用qa-chain.md模板生成三层回答。第一层简要回答（"过拟合是模型在训练数据上表现很好但在新数据上泛化差的现象"），第二层详细推导（展开过拟合的三种原因、检测方法和四种解决方案），第三层延伸思考（提供思考题"为什么增加训练数据可以缓解过拟合？"和推荐学习模型评估方法kp-021）。用户反馈："解释很清楚，特别是检测方法的表格让我一目了然。"

**案例2：复习规划场景**

用户输入："帮我制定人工智能导论的两周复习计划，7月22日考试，每天能学2.5小时。"

系统交互过程：意图路由器识别为review-planner类型 → 检索知识库获取8章45个知识点的结构和权重 → 计算时间分配（总14天×2.5h=35h，按知识点权重和难度分配） → 生成四阶段计划表。用户反馈Day 1计划未完成，系统自动将未完成内容重新分配到剩余天数。用户评价："自适应调整功能很贴心，不会因为一天拖延就全盘混乱。"

## 五、课程设计总结

### 5.1 主要完成工作

通过本次课程设计，完成了以下工作：（1）调研了大模型Agent技术在教育领域的应用现状，论证了选题的时代背景和实践价值；（2）设计了智学助手的四层架构和五大功能模块；（3）实现了3个Agent定义、5个Skill模块、7个提示词模板和45个结构化知识点的知识库；（4）构建了暗色主题的Web演示界面；（5）通过15个测试用例和3名用户评估验证了系统的功能和实用性。

### 5.2 创新点

（1）三层提示词体系（系统级-任务级-工具级），使Agent行为的精细控制成为可能；（2）三层知识结构（索引层+内容层+关系层），支持精准检索和关联推荐；（3）ReAct智能体模式，实现了感知-推理-行动-记忆的完整闭环。

### 5.3 遇到的问题与解决过程

在实现过程中遇到的主要问题包括：PDF/PPTX等二进制文件的文本提取（通过Bash调用unzip和python-docx解决）、提示词版本迭代中行为不一致（通过细化输出格式规范解决）、知识库检索在大量知识点下的效率问题（通过三层索引结构实现O(1)定位解决）。

### 5.4 不足与展望

当前不足：知识库覆盖课程有限（仅2门）；Web界面为模拟演示而非真实API接入；缺少向量检索能力。未来可以在以下方面进行改进：扩展知识库至更多课程（数据结构、操作系统等）、接入真实LLM API实现完整智能对话、引入向量数据库提升语义检索精度、实现多Agent协作架构、增加语音交互和OCR手写笔记识别能力。这些改进将使智学助手从课程设计原型进化为真正实用的学习工具。

### 5.5 心得体会

通过本次课程设计，我从理论到实践深入理解了AI Agent的设计与构建方法。最大的收获是认识到：构建一个真正有用的AI系统，关键不在于使用多么复杂的模型，而在于如何将模型的能力与用户的实际需求精准对接。提示词设计、知识库构建、工具选型这些"工程化"工作，往往比模型本身更决定系统的最终质量。
