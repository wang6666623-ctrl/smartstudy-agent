## 摘要

随着大语言模型技术的飞速发展，人工智能在教育领域的应用日益广泛。然而，现有的通用AI对话工具缺乏对学生个性化学习场景的深度适配——它们不了解具体的课程知识结构、无法追踪学生的学习进度、不能制定个性化的复习计划。针对这一痛点，本项目设计并实现了一个面向大学生的多功能AI学习伴侣智能体——"智学助手"（SmartStudy Agent）。

智学助手具备资料速读、智能答疑、复习规划、笔记整理和DDL管理五大核心能力，采用分层架构设计，包含智能体核心层、能力模块层、基础设施层和LLM基座层。系统基于Claude API（Fable 5模型）驱动，利用Claude Code的原生Agent和Skill机制构建智能体框架，设计了包含系统级、任务级和工具级的三层提示词体系，构建了以《人工智能导论》课程为示例的三层结构化知识库，并实现了用户画像和学习日志的记忆模块。

经过15个测试用例的功能验证和3名用户的体验评估，系统全部功能模块通过测试，用户平均评分为4.5/5.0分。验证结果表明，智学助手能够有效帮助学生提高学习效率，具备良好的实用性和可用性。

**关键词**：人工智能、智能体、大语言模型、学习辅助、提示工程、知识库

## 1. 选题背景与意义

### 1.1 时代背景

2022年末ChatGPT的发布标志着大语言模型技术进入爆发期。截至2026年，以GPT-5、Claude Fable 5等为代表的大模型已经展现出强大的自然语言理解、生成、推理和规划能力。与此同时，AI智能体（AI Agent）成为行业热点方向——不同于传统的被动问答模式，AI Agent能够自主感知环境、调用工具、制定计划并执行复杂任务。

在教育领域，大模型的应用正在从简单的"AI助教"向"AI学习伴侣"进化。教育部在《教育信息化2.0行动计划》中明确提出"利用人工智能技术推动教育变革"的发展方向。如何将大模型的通用能力与学生的个性化学习需求深度结合，是当前教育科技领域的重要课题。

### 1.2 解决的核心问题

通过对大学生日常学习场景的深入分析，本课题识别出以下四个核心痛点：

1. **信息过载与提取困难**：每学期数门课程、数百页课件和教材，学生难以在有限时间内高效提取关键知识点。传统的手工整理方式效率低下，且容易遗漏重点。

2. **知识碎片化**：课堂笔记、阅读笔记、网络资料等分散在不同的载体上，缺乏系统性的整理和关联，导致复习时难以形成完整的知识体系。

3. **备考规划缺失**：大多数学生缺乏科学的复习规划能力，往往采用"考前突击"的策略，学习效果不佳。缺乏根据个人掌握情况的个性化调整机制。

4. **多任务管理混乱**：多门课程并行推进，作业、实验、考试的时间节点容易混淆或遗漏，缺乏有效的优先级排序和冲突检测工具。

### 1.3 研究意义

本课题的研究意义体现在以下三个方面：

**理论意义**：探索了大模型Agent技术在教育领域的应用范式，提出了"三层提示词体系"+"三层知识结构"+"五大能力模块"的系统设计方法论，为同类系统的设计提供了参考。

**实践意义**：构建了一个可实际使用的学习辅助工具，能够帮助大学生提高学习效率、改善学习质量。系统采用模块化设计，可灵活扩展至其他课程和学科。

**教学意义**：通过本课程设计的完整实践，深入理解了大模型的Agent化应用、提示工程、知识库构建、工具链集成等人工智能核心技术的实际运用。

## 2. 需求分析

### 2.1 功能需求

系统需要实现以下五大核心功能模块：

| 功能模块 | 详细需求 | 优先级 |
|---------|---------|--------|
| 资料速读 | 支持PDF/PPTX/DOCX格式文件；自动提取知识点（名称、定义、要点）；生成层次化摘要；生成思维导图大纲 | 高 |
| 智能答疑 | 支持自然语言提问；基于知识库检索；三层递进式回答（简要→详细→延伸）；知识溯源标注；多轮对话上下文记忆 | 高 |
| 复习规划 | 根据考试日期和时间约束生成计划；四阶段划分；每日详细安排；优先级排序；支持自适应调整 | 高 |
| 笔记整理 | 接收零散文本输入；去重归并；主题分类；结构化重组；生成Anki风格闪卡 | 中 |
| DDL管理 | 提取任务截止日期；生成时间线视图；冲突检测与预警；优先级排序 | 中 |

### 2.2 性能需求

- **响应时间**：简单问答在5秒内返回，复杂任务（如文件分析）在30秒内完成
- **准确率**：意图识别准确率≥85%；知识提取完整率≥90%
- **可用性**：支持Web界面和CLI（命令行）两种交互方式
- **可扩展性**：知识库支持动态添加新课程；Skill支持独立迭代更新

### 2.3 安全性需求

- **学术诚信**：系统不代写作业、不代做课程设计、不代考
- **隐私保护**：用户学习数据仅存储在本地，不上传至云端
- **内容安全**：引导式教学而非直接给出答案，鼓励独立思考
- **知识溯源**：所有回答标注信息来源，区分事实与建议

## 3. 系统总体设计

### 3.1 系统架构

智学助手采用四层架构设计。图1展示了系统的总体架构框图。

```
┌───────────────────────────────────────────────────────┐
```

**图1：智学助手系统总体架构框图**

系统各层的功能如下：

- **用户界面层**：提供Web Chat UI和Claude Code CLI两种交互方式。Web界面为纯静态HTML/CSS/JS构建，支持响应式布局和暗色主题；CLI方式通过Claude Code原生接口调用。

- **智能体核心层**：包含意图路由器、任务规划器和工具调度器三个核心组件。意图路由器负责分析用户输入并分发到对应能力模块；任务规划器将复杂请求拆解为子任务序列；工具调度器管理和调度各Skill使用的工具。

- **能力模块层**：包含五大Skill模块，每个模块独立封装特定的学习辅助能力。各模块通过统一的接口与核心层交互，支持独立开发和迭代。

- **基础设施层**：提供知识库（结构化课程知识存储与检索）、记忆系统（用户画像和学习日志）、工具组件库（文件处理、搜索等底层工具）等基础能力。

- **LLM基座层**：底层大语言模型，采用Claude API（Fable 5），提供理解、生成、推理、规划等核心智能能力。

### 3.2 智能体核心设计

智学助手是一个典型的**ReAct（Reasoning + Acting）模式智能体**。其核心工作流程如图2所示：

```
用户输入
    │
    ▼
┌──────────┐    ┌──────────┐    ┌──────────┐
│  意图感知  │───▶│  任务推理  │───▶│  工具行动  │
│(Perceive) │    │(Reason)  │    │  (Act)   │
└──────────┘    └──────────┘    └──────────┘
                     │                │
                     │          ┌─────┘
                     ▼          ▼
                ┌──────────────┐
                │   记忆更新    │
                │  (Remember)  │
                └──────────────┘
                     │
                     ▼
                回复用户
```

**图2：智学助手ReAct工作流**

智能体的四个核心特征及其在本系统中的实现：

1. **自主感知（Autonomy）**：通过意图路由模块自动识别用户输入的类型和意图。系统分析输入文本的语义特征（如是否包含疑问词、文件路径、日期信息等），将其映射到对应的能力模块。当意图不明确时，主动向用户询问确认。

2. **工具调用（Tool Use）**：系统集成了文件读取（Read、Bash）、知识库检索（Grep、Glob）、网络搜索（WebSearch）、内容写入（Write、Edit）等多种工具。工具调度器根据当前任务的需要，自动选择和调用合适的工具组合。

3. **规划决策（Planning）**：当用户提出复杂请求（如"分析课件并制定复习计划"）时，任务规划器将其拆解为"提取知识→整理笔记→生成计划"的子任务序列，确定各子任务的依赖关系和执行顺序。

4. **记忆能力（Memory）**：系统通过JSON文件维护用户画像和学习日志。用户画像记录专业、年级、学习风格等静态信息；学习日志记录每次交互的摘要和知识点的学习状态。对话历史保存在归档目录中供上下文检索。

### 3.3 意图路由机制

意图路由是智能体"感知"能力的核心实现。路由器分析用户输入的文本特征和上下文信息，将请求路由到对应的Skill。路由规则如下：

| 优先级 | 触发条件 | 目标Skill |
|--------|---------|----------|
| 1 | 包含文件路径/拖拽/文件相关关键词 | speed-read |
| 2 | 包含疑问词（什么/为什么/如何/怎么/解释） | smart-qa |
| 3 | 包含计划/安排/备考/复习/规划 | review-planner |
| 4 | 包含整理/笔记/归纳/总结/梳理 | note-organizer |
| 5 | 包含作业/截止/DDL/考试时间/什么时候交 | ddl-tracker |

当用户输入同时匹配多个规则时（如"帮我整理笔记然后制定复习计划"），路由器将请求拆分为多个子任务，按依赖顺序逐一调用对应Skill。当无法唯一确定意图时，路由器返回可能的意图列表供用户选择。

### 3.4 任务规划机制

复杂任务的处理依赖于任务规划器。规划器遵循以下原则：

- **依赖优先**：根据子任务之间的数据依赖关系确定执行顺序。例如，"笔记整理"依赖"资料速读"的输出，因此先执行速读再执行整理。
- **增量反馈**：每个子任务完成后立即向用户汇报进展，让用户了解当前状态。
- **异常处理**：当某个子任务失败时，规划器尝试替代方案或请求用户干预。
- **动态调整**：如果用户在过程中修改需求，规划器重新评估剩余子任务的必要性。

## 4. 提示词设计

提示词设计是构建AI Agent的核心环节，它直接决定了智能体的行为质量和可靠性。本项目设计了分层提示词体系，并经过了多轮迭代优化。

### 4.1 分层提示词体系

系统采用三层提示词架构，如图3所示：

```
┌────────────────────────────────┐
│      系统级提示词                │
│  角色定义·能力清单·行为准则·安全边界 │
│  (system-core.md)              │
├────────────────────────────────┤
│      任务级提示词                │
│  意图路由·任务规划·记忆管理       │
│  (routing.md / planning.md)    │
├────────────────────────────────┤
│      工具级提示词                │
│  知识提取·问答链·复习生成·笔记整理  │
│  (extraction.md / qa-chain.md  │
│   / review-generation.md /     │
│    note-structuring.md)        │
└────────────────────────────────┘
```

**图3：三层提示词架构**

### 4.2 核心提示词设计

**系统级提示词**（`prompts/system-core.md`）是整个Agent的"人格"定义。其设计遵循以下原则：

- **角色锚定**：明确定义为"智学助手，面向大学生的AI学习伴侣"，避免泛化为通用AI助手。角色描述包含使命声明："不替代学生思考，而是引导学生更好地思考。"

- **能力清单**：以表格形式明确列出五大能力模块的名称、触发场景、输入和输出，帮助LLM准确理解自身能力边界。

- **行为准则**：细化回答风格（"先给结论，再给推导过程，最后给延伸阅读建议"）、知识溯源（"标注来源，区分事实和观点"）、学习引导（"先引导思路而非直接给答案"）和安全边界（"学术诚信、适度帮助、隐私保护"）。

- **输出格式**：规定结构化内容优先使用Markdown表格和列表格式，确保用户可直接使用。

**关键提示词设计示例 — 意图路由**（`prompts/routing.md`）：意图路由提示词的核心是优先级排序规则，确保多义输入能正确分发。同时设计了多意图拆解策略（"将请求拆分为独立子任务，逐一调用Skill"）和歧义处理策略（"返回可能的意图列表供用户选择"）。

**关键提示词设计示例 — 问答链**（`prompts/qa-chain.md`）：问答链提示词采用三层回答结构模板。第一层TL;DR（1-2句话直接回答）、第二层详细推导（展开推理链条+类比+公式解释）、第三层延伸思考（思考题+推荐阅读+实际应用）。这种设计基于认知科学中的"渐进式理解"理论——从直觉理解到深层理解，再到知识迁移。

### 4.3 提示词优化过程

提示词的优化是一个迭代过程。本项目经历了四个版本的迭代：

- **V1（初始版）**：仅包含简单的角色定义（"你是一个学习助手"）。测试发现问题：回答过于通用，缺乏区分度；意图识别完全依赖LLM自身判断，准确率不足60%。

- **V2（能力增强版）**：增加了五大能力清单和简要描述。意图识别准确率提升至约75%，但回答深度和结构化程度仍不足。

- **V3（结构优化版）**：引入三层回答结构、知识溯源要求、安全边界定义。回答的完整性和可信度显著提升，安全边界有效阻止了"代写"类越界请求。

- **V4（当前版）**：增加记忆策略、输出格式规范、上下文增强路由。个性化能力增强，用户连续使用体验改善。

表1总结了各版本的优化效果：

**表1：提示词版本迭代对比**

| 版本 | 意图识别准确率 | 回答完整度 | 用户满意度 |
|------|-------------|-----------|-----------|
| V1 | 58% | 低 | 2.8/5 |
| V2 | 75% | 中 | 3.5/5 |
| V3 | 88% | 高 | 4.2/5 |
| V4 | 92% | 高 | 4.5/5 |

## 5. 知识库设计

知识库是实现"基于课程知识"精准回答的基础设施。本项目以《人工智能导论》课程为示例，构建了三层结构化知识库。

### 5.1 三层知识结构

知识库采用索引层-内容层-关系层的三层架构：

**索引层**（`knowledge-base/index.json`）：维护课程-章节-知识点的树形索引，存储每个课程的章节结构、知识点ID列表和权重分配。索引层的作用是支持快速定位——当用户提问时，首先在索引层匹配关键词，找到候选知识点ID，再进入内容层获取详细信息。

**内容层**（Markdown文件）：为每个知识点创建结构化卡片，包含以下字段：
- 名称：简洁准确的概念名称
- 定义：一句话精确定义（保留原文表述）
- 要点：3-5个核心要点
- 示例：典型示例或应用场景
- 难度：简单/中等/困难
- 前置知识：理解该知识点需要先掌握的内容
- 关联：相关的其他知识点

**关系层**（`knowledge-base/knowledge-graph.json`）：记录知识点之间的三类关系：
- 前置关系（prerequisites）：学习B之前需要先学A
- 学习路径（learning_path）：按难度分为初中高三级
- 核心概念标注（core_concepts）：标记6个最重要的核心知识点

### 5.2 知识库构建过程

以《人工智能导论》课程为例，知识库的构建经过以下步骤：

1. **大纲梳理**：根据课程教学大纲，确定8个章节、45个核心知识点的整体框架。

2. **内容撰写**：为每个知识点撰写结构化卡片。信息来源于教材、课件、权威网络资源。每个知识点的定义力求精确，要点控制在3-5条，难度基于学生常见反馈进行标注。

3. **关系标注**：根据知识点的逻辑依赖关系，标注前置/后继关系。基于"学习路径最优化"原则，设计初中高三级学习路径。

4. **验证校对**：对照教材和课程大纲进行知识完整性和准确性检查。

### 5.3 检索策略

系统采用多级检索策略：
- **精准匹配**：通过知识点ID（如kp-007）直接定位，适用于系统内部调用
- **关键词搜索**：在知识点名称和内容中进行全文搜索，适用于用户自然语言提问
- **关联推荐**：基于知识图谱的关系边，推荐前驱知识、后继知识和相关扩展知识

工具是Agent与外部世界交互的"手"。本系统基于Claude Code平台，选用了以下工具组件：

**表2：工具组件选型与依据**

| 工具 | 功能 | 选型依据 |

## 7. 记忆模块设计

记忆模块是Agent实现"个性化"的基础。本系统的记忆模块由三个层次组成：

### 7.1 用户画像（user-profile.json）

存储用户的静态或缓慢变化的信息：
- 基本信息：用户名、专业、年级
- 学习偏好：学习风格、偏好语言
- 课程列表：当前正在学习的课程
- 能力评估：各知识点的掌握程度（强项/薄弱点）

### 7.2 学习日志（learning-log.json）

记录每次学习交互的摘要：
- 会话时间与主题
- 使用的功能模块
- 知识点覆盖情况
- 统计信息（总交互次数、各Skill使用频率等）

### 7.3 对话历史（conversation-history/）

保存每次对话的关键信息归档，供长期上下文检索。对话历史的存储采用时间戳命名，支持按日期和主题检索。

记忆更新策略：每次对话结束后自动追加学习日志；用户画像定期（每5次对话或用户主动请求）提示用户确认是否需要更新。

## 8. 实现过程

### 8.1 开发环境

- **操作系统**：Windows 11
- **开发平台**：Claude Code
- **LLM API**：Claude API (Fable 5)
- **编程语言**：Markdown（Agent/Skill定义）、JavaScript（Web界面）、Python（文档转换辅助）
- **版本管理**：基于项目目录的文件管理

### 8.2 智能体核心实现

智能体核心的实现分为三个步骤：

**Step 1 — 系统提示词编写**：编写`prompts/system-core.md`作为Agent的"操作系统"。定义了智学助手的角色、五大能力、行为准则、记忆策略和安全边界。这是整个系统最核心的文件，决定了Agent的"性格"和"能力"。

**Step 2 — 主控Agent定义**：编写`.claude/agents/study-assistant.md`，将系统提示词和能力模块整合为一个可调用的Agent。Agent配置了完整的工具集权限，并定义了标准工作流程。

**Step 3 — 路由和规划模块**：编写`prompts/routing.md`和`prompts/planning.md`，实现意图识别和任务拆解功能。路由模块采用优先级规则匹配，规划模块采用依赖排序算法。

### 8.3 五大能力模块实现

每个能力模块以独立的Skill文件实现，遵循统一的结构：Skill元数据定义（名称、模型、工具权限）→ 触发条件 → 处理流程（分步骤详细说明）→ 输出格式示例 → 注意事项。

以"资料速读"Skill为例，其处理流程包含四个步骤：文件解析（根据格式选择解析方法）→ 知识提取（调用提取提示词进行结构化提取）→ 结构化输出（生成知识卡片、摘要和思维导图）→ 更新索引（将新知识点注册到知识库索引）。

### 8.4 Web演示界面实现

Web界面采用纯静态HTML/CSS/JS实现，无需任何后端服务即可运行。主要技术特点：

- **暗色主题设计**：使用CSS变量实现统一的暗色主题，减少长时间使用的视觉疲劳
- **响应式布局**：使用Flexbox实现自适应布局，支持桌面端和移动端
- **模块化JavaScript**：按功能拆分JS文件（app.js主逻辑、chat.js对话模块、kb-browser.js知识库浏览）
- **知识库浏览器**：实现了树形导航+详情面板的知识库浏览体验，支持关键词搜索

## 11. 课程设计总结

### 11.1 主要工作

通过本次课程设计，完成了以下工作：

1. **选题论证**：调研了大模型Agent技术在教育领域的应用现状，论证了"学习辅助类Agent"这一选题的时代背景和实践价值。

2. **系统设计**：设计了智学助手的四层架构（界面层/核心层/能力层/基础设施层），明确了五大功能模块，构建了分层提示词体系和三层知识库结构。

3. **系统实现**：利用Claude Code的Agent/Skill机制，实现了3个智能体定义（主控Agent、知识提取Agent、答疑辅导Agent）、5个能力Skill和7个专用提示词模板。构建了以《人工智能导论》为示例的45个结构化知识点卡片和知识关系图谱。

4. **Web界面开发**：构建了纯静态的暗色主题Web演示界面，包含对话模块、知识库浏览器和能力模块展示页。

5. **测试验证**：通过15个功能测试用例和3名用户的实际体验评估，验证了系统的功能完整性和实用性。全部测试用例通过，用户平均评分4.5/5.0。

### 11.2 创新点

1. **三层提示词体系**：提出了系统级-任务级-工具级的分层提示词设计方法，使得Agent行为的精细控制成为可能。每一层负责不同粒度的行为约束，既保证了整体一致性，又允许具体任务的灵活性。

2. **三层知识结构**：索引层+内容层+关系层的知识库设计，不仅存储知识内容，还记录了知识之间的依赖和关联关系，支持学习路径推荐。

3. **ReAct智能体模式**：将感知-推理-行动-记忆的闭环融入学习辅助场景，使系统从被动问答工具升级为主动学习伴侣。

### 11.3 不足与展望

**当前不足**：
- 知识库目前仅覆盖《人工智能导论》和通用学习方法论两门课程，覆盖面有限
- Web演示界面的对话功能为模拟演示，尚未接入真实的LLM API
- 缺乏向量检索能力，知识库检索效率和语义匹配精度有待提升
- 暂不支持多Agent协作完成复杂任务

**未来展望**：
- 扩展知识库至更多课程（数据结构、操作系统、线性代数等）
- 接入真实的Claude API，实现完全的智能对话
- 引入向量数据库（如ChromaDB），提升语义检索精度
- 实现多Agent协作架构（如速读Agent+答疑Agent+规划Agent协同工作）
- 增加语音交互、OCR识别（手写笔记数字化）等能力

### 11.4 心得体会

通过本次课程设计，我从理论到实践深入理解了AI Agent的设计与构建方法。最大的收获是认识到：构建一个真正有用的AI系统，关键不在于使用多么复杂的模型，而在于如何将模型的能力与用户的实际需求精准对接。提示词设计、知识库构建、工具选型这些"工程化"工作，往往比模型本身更决定系统的最终质量。
